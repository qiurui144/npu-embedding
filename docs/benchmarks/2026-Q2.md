## Methodology

### 1. Deterministic metrics (W4 J6, ships in v0.6)

For each query in `rust/tests/golden/queries.json`:

- **Hit@K**: 1 if any `acceptable_hits[]` ID appears in the top-K results, else 0.
- **Recall@K**: `|hits in top-K| / |acceptable_hits|`.
- **MRR (Mean Reciprocal Rank)**: `1 / rank` of the first hit (0 if none).

These are computed in pure Rust (no LLM dependency, fully reproducible). The unit tests
in `rag_quality_benchmark.rs` verify the metric implementations on fixed inputs.

### 2. LLM-judged metrics (W5 J6, planned for v0.6.1+)

Following the [RAGAS framework](https://github.com/explodinggradients/ragas):

- **Faithfulness**: Does the answer make claims supported by the retrieved context?
- **Answer Relevancy**: Does the answer address the user's question?
- **Context Precision**: Are the relevant chunks ranked first in the retrieved context?
- **Context Recall**: Does the retrieved context cover the ground-truth answer?

These require a reliable LLM judge (typically GPT-4-class). We defer the integration to
v0.6.1 once we've stabilized the LLM provider abstraction (W5 C4 multi-LLM router).

### 3. Real-corpus harness

Mock-corpus tests live in `rag_quality_benchmark.rs` and run on every PR. Real-corpus
benchmarks against the locked corpora below are gated behind `#[ignore]` and triggered
manually:

```bash
cd rust
cargo test --release -p attune-core --test rag_quality_benchmark -- --ignored
```

This avoids burning CI minutes on a 30-minute index-and-query run for every PR.

## Comparison framework (vs anything-llm / khoj / quivr)

To make the numbers meaningful, we will publish side-by-side comparisons on the same corpora:

| Tool | Embedding | Reranker | Chunking | Retrieval |
|------|-----------|----------|----------|-----------|
| Attune v0.6 | bge-m3 (Ollama) | none (RRF) | semantic + path prefix (J1) | hybrid BM25 + vector with explicit min_score (J3) |

The v0.6.0 GA benchmark PR will add the actual numbers + the run scripts for each
competitor.

## Why this report ships in v0.6 even without final numbers

Per W4 plan J6, the **methodology + framework + reproducibility** are the v0.6 deliverables.
Baseline numbers will land in the GA release after a production benchmark run on real
corpora — they are not blocking v0.6.0-rc.1.

This is intentional: it forces us to lock the methodology (which queries, which corpora,
which K, which metrics) **before** seeing the numbers, preventing the well-known trap of
"goalpost shifting after the data comes in."

## How to reproduce

1. Clone Attune at the v0.6.0-rc.1 tag.
2. Pin and download corpora per `_corpus_pins`:
   ```bash
   bash scripts/download-corpora.sh
   ```
3. Run the benchmark:
   ```bash
   cd rust
   cargo run --release --bin attune-server &  # background
   sleep 5
   curl -X POST http://localhost:18900/api/v1/index/bind -d '{"path": "../tests/corpora/rust-book"}'
   # wait for indexing to complete (watch /api/v1/status)
   cargo test --release -p attune-core --test rag_quality_benchmark -- --ignored --nocapture
   ```
4. Compare your output to the published baseline.

## Honest limitations

- **No LLM-judged metrics in v0.6** (Faithfulness/AR/CP/CR). The deterministic metrics
  cover retrieval quality; they do not catch hallucinations in the generated answer.
  Bridging this is the v0.6.1 W5 J6 priority.
- **Single-language scenarios per corpus** — mixed-language queries (e.g. legal terms in
  English asking Chinese law) need a separate scenario which we'll add in v0.7.
- **No latency benchmarks yet** — quality first, latency in v0.7 when we add the PromptIQL
  cache layer.
